# Database Schema & Data Architecture (MVP)

This document details the data structures, database schemas, and data flow mechanisms for the Unified Design Thinking Workbench MVP (Max 3 concurrent users, GCP `europe-west9` deployment).

## 1. Database Paradigm
The architecture relies on three distinct databases to manage state, relationships, and semantic search:
1.  **Cloud SQL (PostgreSQL)**: Handles relational business logic (Users, Projects/Workspaces, Sessions, Permissions).
2.  **Neo4j Aura (Graph)**: Manages the topological relationships of the Design Thinking methodology (e.g., *Insight X* -> `DERIVES` -> *HMW Y* -> `INSPIRES` -> *Concept Z*).
3.  **Vertex AI Vector Search**: Handles semantic similarity matching for unstructured text (Transcripts, Field Observations, generated Notes).

---

## 2. Relational Schema (Cloud SQL / PostgreSQL)
The relational DB manages the rigid, tabular data required to authorize users and organize workspaces.

### Table: `users`
*   `id` (UUID, Primary Key)
*   `email` (String, Unique)
*   `role` (Enum: `FACILITATOR`, `PARTICIPANT`)
*   `created_at` (Timestamp)

### Table: `projects`
*   `id` (UUID, Primary Key)
*   `name` (String)
*   `facilitator_id` (UUID, Foreign Key -> `users.id`)
*   `current_gate` (Enum: `0_PREP`, `1_INSPIRATION`, `2_IDEATION`, `3_IMPLEMENTATION`)
*   `current_phase` (Int: `0` to `9`)
*   `notion_root_url` (String) - *Configuration for MCP exports.*
*   `miro_board_id` (String) - *Configuration for MCP exports.*
*   `created_at` (Timestamp)

### Table: `sessions`
*   `id` (UUID, Primary Key)
*   `project_id` (UUID, Foreign Key -> `projects.id`)
*   `yjs_room_id` (String) - *Identifier for the real-time syncing server.*
*   `is_active` (Boolean)

---

## 3. Graph Data Model (Neo4j Aura)
Neo4j ensures that every generated artifact maintains cryptographic lineage back to raw user observations, preventing algorithmic hallucination.

### Key Node Labels
*   **`Project`**: Root node for traversal.
*   **`InterviewProfile`**: Metadata node (Name, Age, Profession, Location).
*   **`Observation`**: A distinct, raw qualitative data point (derived from a Transcript or Field Log).
*   **`Insight`**: Synthesized understanding (Phase 3).
*   **`HMW`**: "How Might We" statement (Phase 3).
*   **`Concept`**: Brainstormed idea (Phase 4).
*   **`Prototype`**: Wireframe/Storyboard (Phase 5).

### Key Relationships (Edges)
*   `(:Observation)-[:BELONGS_TO]->(:InterviewProfile)`
*   `(:Insight)-[:SYNTHESIZED_FROM]->(:Observation)`
*   `(:HMW)-[:ADDRESSES]->(:Insight)`
*   `(:Concept)-[:SOLVES]->(:HMW)`
*   `(:Prototype)-[:MANIFESTS]->(:Concept)`
*   `(:Observation)-[:MAPPED_IN]->(:Project)`

### Example Cypher Query (Lineage Tracing)
*When a Facilitator asks Domo "Why did we build this prototype feature?", Domo executes:*
```cypher
MATCH (p:Prototype {id: "123"})-[:MANIFESTS]->(c:Concept)-[:SOLVES]->(h:HMW)-[:ADDRESSES]->(i:Insight)-[:SYNTHESIZED_FROM]->(o:Observation)
RETURN o.text AS RawContext
```
*(Note: For generic Field Observations not tied to a specific person, the `Observation` node connects directly via `[:MAPPED_IN]->(:Project)` rather than an `InterviewProfile`.)*

## 4. Semantic Schema (Vertex AI Vector Search)
Vertex AI handles unstructured matching (e.g., "Find all times a user complained about the login screen"). Only the raw text, project context, and vector embeddings are stored here.

### Vector Payload Structure
*   `id` (UUID) -> *Maps 1:1 with the `id` of the corresponding Node in Neo4j.*
*   `project_id` (UUID) -> *Critical for sandboxing semantic searches to the **current project only**.*
*   `text_chunk` (String) -> *The raw text (max ~500 tokens).*
*   `embedding` (Array[Float]) -> *Generated by `text-embedding-004`.*
*   `node_type` (String) -> *e.g., "Observation", "Insight", "HMW".*

---

## 5. The Dual-Write Sync Flow & PII Sanitization
To ensure the graph and vector databases remain perfectly synchronized during ingestion from the Companion App:

1.  **Ingestion**: Audio is recorded in the Companion app -> Sent to GCP Cloud Run (Python Whisper Worker).
2.  **Transcription**: Speech converted to text.
3.  **Sanitization (Cloud DLP)**: Text passes through GCP Cloud DLP.
    *   *Mechanism*: Identifies PII (e.g., "Kieran").
    *   *Mutation*: Replaces with synthetic token `[PERSON_1]`.
    *   *Mapping Storage*: Stores `{ "Kieran": "[PERSON_1]" }` in Google Memorystore (Redis) pegged to the current `session_id`. *(For the MVP, this mapping is explicitly **ephemeral** and per-session. A permanent secure lookup table is deferred to future enterprise versions.)*
4.  **Notion Export**: The *sanitized* full transcript is sent via MCP to the Notion Database defined in the `projects` table.
5.  **Chunking**: Text is split into semantic observations.
6.  **Dual-Write Transaction**:
    *   A) Embeddings generated via Vertex AI -> Inserted into Vector Search index (tagged with `project_id`).
    *   B) Nodes/Edges generated -> Inserted into Neo4j graph.
    *   *Constraint*: Both database insertions must succeed, or the entire transaction rolls back to prevent orphaned semantic data without topological context.
